```python
def three_sum(nums):
    n = len(nums)
    if not nums or n < 3:
        return []
    nums.sort()
    res = []
    for i in range(n - 2):
        if nums[i] > 0:
            break
        if i > 0 and nums[i] == nums[i - 1]:
            continue
        left = i + 1
        right = n - 1
        while left < right:
            total = nums[i] + nums[left] + nums[right]
            if total == 0:
                res.append([nums[i], nums[left], nums[right]])
                while left < right and nums[left] == nums[left + 1]:
                    left += 1
                left += 1
                while left < right and nums[right] == nums[right - 1]:
                    right -= 1
                right -= 1
            elif total > 0:
                right -= 1
            else:
                left += 1
    return res


nums = [-1,0,1,2,-1,-4] # [[-1,-1,2],[-1,0,1]]
result = three_sum(nums)
print(result)
```

三数之和（3Sum）是 LeetCode 中极具代表性的题目。如果说“两数之和”是**寻找另一半**的单身派对，那么“三数之和”就是一场**寻找稳固三角（小团体）**的复杂社交活动。

它的核心思想是：**“先固定一个领头人，再让另外两个人在剩下的队伍里‘掐头去尾’精准匹配。”**


### 一、 核心思想：从“盲目乱撞”到“有序缩小”

如果用暴力法 $O(N^3)$，就像是在人群中随机抓三个人问“你们三个认识吗？”，效率极低。
三数之和的精髓在于 **“排序 + 双指针”**。

1.  **秩序感（排序）**：首先让所有人按个头（数值）排好序。有了顺序，我们才知道“太大了”往左走，“太小了”往右走。
2.  **化繁为简**：把“三数之和”降维打击。固定一个数 $a$，问题就变成了：在剩下的有序队伍里，找两个数 $b$ 和 $c$，使得 $b + c = -a$。这本质上就是**有序版的两数之和**。

---

### 二、 算法逻辑：三位主角的配合

想象我们在一个排好序的数组 `nums` 里进行搜索：

#### 1. 固定“领头人” (指针 $i$)
我们从左到右遍历数组。每一个被选中的 `nums[i]` 都是当前“三角关系”的老大。
* **去重预警**：如果当前的老大跟前一个老大长得一模一样，直接跳过（为了避免产生重复的三元组）。
* **悲观预警**：如果老大本人已经大于 0 了，因为数组是升序的，他后面的小弟肯定也大于 0，三个人加起来绝对不可能等于 0。**直接散伙！**

#### 2. 双指针“掐头去尾” (指针 $L$ 和 $R$)
老大确定后，他在剩下的右侧区间里招募另外两个成员。
* **左指针 $L$**：从老大的右手边第一个人（$i+1$）开始。
* **右指针 $R$**：从队伍的最末尾（$n-1$）开始。

#### 3. 动态调整（重点！）
计算三人的力量总和 `sum = nums[i] + nums[L] + nums[R]`：
* **`sum == 0`**：**完美匹配！** 记录下这三个人。然后 $L$ 和 $R$ 要同时向内移动，去寻找下一组可能的搭档。注意，移动时如果遇到长得一样的数字，也要跳过。
* **`sum < 0`**：说明大家还是太弱了（数值太小）。为了变强，我们需要让个头更大的加入，所以 **$L$ 右移**。
* **`sum > 0`**：说明大家太强了（数值太大）。为了减弱，我们需要换掉那个最大的，所以 **$R$ 左移**。

---

### 三、 过程模拟 (`nums = [-1, 0, 1, 2, -1, -4], target = 0`)

**第一步：排序**
数组变为：`[-4, -1, -1, 0, 1, 2]`

**第二步：开始遍历**
1.  **$i$ 指向 -4**：
    * $L$ 指向 -1，$R$ 指向 2。
    * `-4 + (-1) + 2 = -3 < 0`。$L$ 右移。
    * ...直到 $L$ 和 $R$ 相遇也没找到。
2.  **$i$ 指向 -1** (第一个 -1)：
    * $L$ 指向下一个 -1，$R$ 指向 2。
    * `-1 + (-1) + 2 = 0`。**【匹配成功：[-1, -1, 2]】**。
    * $L$ 移到 0，$R$ 移到 1。
    * `-1 + 0 + 1 = 0`。**【匹配成功：[-1, 0, 1]】**。
3.  **$i$ 指向 -1** (第二个 -1)：
    * **发现跟前一个老大一样！跳过。**（防止重复收集 `[-1, 0, 1]`）

---

### 四、 为什么不用哈希表？

虽然“两数之和”哈希表很快，但在“三数之和”中：
1.  **去重极其困难**：哈希表很难处理 `[-1, 0, 1]` 和 `[0, -1, 1]` 这种重复解，往往需要额外的排序或集合操作。
2.  **空间成本**：排序+双指针的空间复杂度是 $O(1)$（或 $O(\log N)$ 取决于排序算法），比哈希表更省钱。

---

### 五、 复杂度分析

* **时间复杂度：$O(N^2)$**
    * 排序需要 $O(N \log N)$。
    * 外层循环跑 $N$ 次，内层双指针也跑 $N$ 次。$N \times N = N^2$。
* **空间复杂度：$O(1)$**（忽略排序开销）
    * 我们只用了几个指针，没有额外的大账本。

---

### 六、 总结金句

> **“三数之和就像是精准的天平实验：排序定基准，固定一端（i），两端动态加减（L 和 R）。每一个匹配都是在寻找平衡，每一个跳过都是在拒绝平庸（重复）。”**
