
[14. Longest Common Prefix](https://leetcode.cn/problems/longest-common-prefix)

[0014_longest_common_prefix](./html/0014_longest_common_prefix.html ":include :type=iframe") 

```python
def longest_common_prefix(strs):
    if not strs:
        return ""
    first_str = strs[0]
    for i in range(len(first_str)):
        for other_str in strs[1:]:
            if i >= len(other_str) or first_str[i] != other_str[i]:
                return first_str[:i]
    return first_str


strs = ["flower","flow","flight"] # "fl"
result = longest_common_prefix(strs)
print(result)
```


这道题目（LeetCode 14. 最长公共前缀）的核心思想是 **“垂直扫描（Vertical Scanning）”**。如果说前面的题目是在做“分拣”或“收割”，这道题更像是在做一个 **“多方比对的质量检测”**。

你可以把这个算法想象成一个 **“找茬游戏”**，只不过我们要找的是大家**共同拥有**的部分。

---

### 一、 核心思想：步调一致，少一个都不行

最长公共前缀的要求非常苛刻：这个前缀必须在**所有**字符串中都一模一样。

1.  **标杆思维**：我们先选定第一个字符串 `strs[0]` 作为“基准标杆”。
2.  **逐位比对**：我们不是一个单词一个单词地看，而是**一列一列地看**。看所有单词的第一位是不是一样，再看第二位……
3.  **熔断机制**：只要发现任何一个单词在某一位上跟不上节奏（字符不同，或者单词已经到底了），比对立即结束，之前一致的部分就是我们要的答案。

---

### 二、 算法逻辑：垂直扫描

以 `strs = ["flower", "flow", "flight"]` 为例：

#### 1. 选定标杆
拿第一个单词 `"flower"` 出来，我们准备一位一位地检查它的字符。

#### 2. 第一轮比对：第 0 位
* 基准字符是 `'f'`。
* 检查 `"flow"`：第 0 位是 `'f'`，过。
* 检查 `"flight"`：第 0 位是 `'f'`，过。
* **结论**：目前公共前缀是 `"f"`。

#### 3. 第二轮比对：第 1 位
* 基准字符是 `'l'`。
* 检查 `"flow"`：第 1 位是 `'l'`，过。
* 检查 `"flight"`：第 1 位是 `'l'`，过。
* **结论**：目前公共前缀是 `"fl"`。

#### 4. 第三轮比对：第 2 位
* 基准字符是 `'o'`。
* 检查 `"flow"`：第 2 位是 `'o'`，过。
* 检查 `"flight"`：第 2 位是 `'i'`。**报警！不匹配！**
* **结果**：立即返回标杆的前 2 位，即 `"fl"`。



---

### 三、 过程模拟 (`strs = ["flower", "flow", "flight"]`)

1.  **i = 0**: 基准 'f'。遍历其他词，全有 'f'。继续。
2.  **i = 1**: 基准 'l'。遍历其他词，全有 'l'。继续。
3.  **i = 2**: 基准 'o'。
    * 检查 `"flow"`：有 'o'。
    * 检查 `"flight"`：是 'i'，`'o' != 'i'`。
    * **熔断**：返回 `first_str[:2]` 即 `"fl"`。

---

### 四、 复杂度分析

* **时间复杂度：$O(S)$**
    * $S$ 是所有字符串中字符的数量总和。最坏情况下（所有单词完全一样），我们会把每个字符都看一遍。
* **空间复杂度：$O(1)$**
    * 我们只用了常数级别的额外空间（几个索引变量），没有开辟新的哈希表或大数组。

---

### 五、 总结金句

> **“垂直扫描就像是阅兵式上的排队看齐。指挥官盯着第一排的每一个人，只要发现后面哪一列有人没跟上或者掉队了，队伍就只能排到那儿为止。这种‘一人出错，全家收工’的逻辑，保证了寻找前缀的高效性。”**
