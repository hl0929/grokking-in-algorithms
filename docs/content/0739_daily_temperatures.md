```python
def daily_temperatures(temperatures):
    n = len(temperatures)
    ans = [0] * n
    stack = []
    for i in range(n):
        while stack and temperatures[i] > temperatures[stack[-1]]:
            j = stack.pop()
            ans[j] = i - j
        stack.append(i)
    return ans


temperatures = [73,74,75,71,69,72,76,73] # [1,1,4,2,1,1,0,0]
result = daily_temperatures(temperatures)
print(result)
```

这道题目（LeetCode 739. 每日温度）的核心思想是 **“单调递减栈（Monotonic Decreasing Stack）”**。它是处理“寻找下一个更大元素”这类问题的标准模板。

如果说“有效的括号”是匹配符号，那么这道题就是在寻找 **“未来的救星”**。

---

### 一、 核心思想：等一个暖阳

想象你站在一排温度计前：
1.  **寒冬潜伏**：如果今天的温度比昨天低，或者和昨天一样，你只能继续等待，把昨天的坐标记在心里。
2.  **气温回升**：一旦哪天温度升高了（当前温度 > 栈顶温度），这个“暖阳”就拯救了栈里所有比它冷的那些日子。
3.  **计算跨度**：每拯救一个日子，我们就计算一下“今天”和“那个日子”相隔了几天。



---

### 二、 算法逻辑：保持栈的单调性

#### 1. 变量准备
* `ans`：结果数组，默认全为 `0`。如果某个日子等不到更高的温度，它就永远是 `0`。
* `stack`：**重点！里面存的是下标（Index）**，而不是具体的温度。通过下标，我们既能找到温度，又能计算天数差。

#### 2. 维护规则
* 我们遍历每一天的温度：
    * **当“今天”比“栈顶那天”暖和时**：
        * 栈顶那天终于等到了！把它弹出来（`pop`）。
        * 计算距离：`今天下标 i - 过去下标 j`，存入 `ans[j]`。
        * **继续检查**：可能今天的温度很高，能一口气拯救好几个还在栈里苦等的日子（`while` 循环）。
    * **入栈**：不管今天有没有救别人，今天最后都要入栈，等待它的“未来救星”。

#### 3. 栈的形状
* 栈内留下的下标对应的温度，一定是 **单调递减** 的。因为一旦出现一个高的，就把前面低的都踢走了。

---

### 三 : 过程模拟 (`[73, 74, 75, 71, 69, 72]`)

1.  **73 入栈**：`stack = [0]` (下标)
2.  **74 来了**：74 > 73！
    * 弹出 0，`ans[0] = 1 - 0 = 1`。
    * 74 入栈：`stack = [1]`。
3.  **75 来了**：75 > 74！
    * 弹出 1，`ans[1] = 2 - 1 = 1`。
    * 75 入栈：`stack = [2]`。
4.  **71, 69 陆续入栈**：因为它们都比 75 小。`stack = [2, 3, 4]`。
5.  **72 来了**：
    * 72 > 69：弹出 4，`ans[4] = 5 - 4 = 1`。
    * 72 > 71：弹出 3，`ans[3] = 5 - 3 = 2`。
    * 72 < 75：停！75 还没等到救星。
    * 72 入栈：`stack = [2, 5]`。

---

### 四、 复杂度分析

* **时间复杂度：$O(N)$**
    * 别看有 `while` 循环，但每个日期的下标最多只进栈一次、出栈一次。总操作数和 $N$ 成线性关系。
* **空间复杂度：$O(N)$**
    * 最坏情况下（温度一路走低），栈里会存满所有的下标。

---

### 五、 总结金句

> **“这叫‘单调守望，一触即发’。单调栈最优雅的地方在于它只保留‘有价值的悬念’。每一个被压入栈的下标，都是一个尚未兑现的承诺。而每一次 `pop`，都是对过去漫长等待的一个完美交代。通过维持栈内元素的单调性，我们把 $O(N^2)$ 的暴力搜索压缩成了 $O(N)$ 的电光火石。”**
